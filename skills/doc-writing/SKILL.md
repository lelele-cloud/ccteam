# 文档编写技能

系统化编写技术文档，确保文档清晰、准确、易于理解和维护。涵盖从信息收集到文档发布的完整流程。

## 适用角色

- 文档工程师（主要）
- 所有开发角色（参考）

## 输入/输出

**输入:** 技术资料、代码、人员访谈、现有文档、用户反馈

**输出:**
- 技术文档: `docs/ccteam/10-docs/`
- API文档: `docs/ccteam/05-api/`
- 用户手册: `docs/ccteam/10-docs/user-guide/`
- README: 项目根目录

## 工作流程

```
文档规划 → 信息收集 → 内容撰写 → 评审修订 → 发布维护
```

### 步骤1: 文档规划

确定文档类型和结构：

| 类型 | 目的 | 受众 | 特点 |
|------|------|------|------|
| 概述 | 快速了解 | 所有人 | 简洁、宏观 |
| 教程 | 学习使用 | 新手 | 步骤详细、有示例 |
| 参考 | 查询细节 | 开发者 | 完整、准确 |
| 故障排查 | 解决问题 | 运维/用户 | 问题导向 |

### 步骤2: 信息收集

- 阅读源代码和注释
- 运行和测试功能
- 访谈开发者和产品经理

访谈问题清单见 [references/interview-questions.md](references/interview-questions.md)

### 步骤3: 内容撰写

**写作原则:**
- 清晰: 一句话一个意思，使用主动语态
- 简洁: 删除冗余词汇，直接切入主题
- 准确: 技术细节准确，代码示例可运行
- 一致: 术语统一，格式统一

**代码示例要求:**
- 完整可运行
- 有必要的注释
- 包含预期输出

### 步骤4: 评审修订

1. 自我检查（流畅性、链接、格式）
2. 请开发者检查技术准确性
3. 请新人检查可理解性
4. 根据反馈修改

### 步骤5: 发布维护

- 定期检查时效性
- 收集用户反馈
- 及时更新内容

## 质量检查

**内容检查:**
- [ ] 信息准确完整
- [ ] 示例可运行
- [ ] 链接有效

**结构检查:**
- [ ] 大纲清晰
- [ ] 层级合理
- [ ] 目录完整

**写作检查:**
- [ ] 表述清晰
- [ ] 术语一致
- [ ] 无错别字

## 常见陷阱

| 陷阱 | 规避方法 |
|------|----------|
| 信息收集困难 | 代码是一手资料 |
| 内容快速过时 | 与代码同仓库，CI检查 |
| 示例不可运行 | 自动化测试示例代码 |
| 用户找不到信息 | 改进搜索和导航 |
