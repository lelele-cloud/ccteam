# PRD编写技能

## 概述

编写专业的产品需求文档（Product Requirements Document），确保需求的完整性、清晰性和可执行性，为开发团队提供明确的实现指导。

## 适用角色

- 产品经理（主要）
- 项目经理
- 业务分析师

## 输入

- **需求分析报告**：已完成的需求分析结果
- **业务目标**：项目的业务目标和成功指标
- **技术约束**：已知的技术限制和依赖
- **设计资源**：原型图、UI设计稿（如有）

## 输出

- **PRD文档**：`docs/ccteam/01-requirements/prd.md`
- **功能规格说明**：各功能模块的详细规格
- **数据字典**：核心数据实体定义

## 执行步骤

### 步骤1：准备工作
1. 收集所有输入材料
2. 确认需求分析报告已完成
3. 与干系人确认PRD范围

### 步骤2：撰写概述部分
1. 清晰描述项目背景
2. 定义可衡量的目标（SMART原则）
3. 明确范围边界
4. 建立术语表

### 步骤3：用户分析
1. 定义目标用户群体
2. 创建用户画像（至少2-3个）
3. 绘制用户旅程图

### 步骤4：功能需求详述
1. 按优先级组织功能模块
2. 为每个功能编写：
   - 用户故事
   - 功能流程
   - 业务规则
   - 数据说明
   - 验收标准
   - 异常处理

### 步骤5：非功能需求
1. 定义性能指标（具体数值）
2. 明确安全要求
3. 说明可用性和兼容性要求

### 步骤6：数据与接口
1. 设计数据模型
2. 编写数据字典
3. 列出接口需求

### 步骤7：实施计划
1. 制定里程碑
2. 识别依赖关系
3. 评估风险

### 步骤8：评审与修订
1. 内部自查
2. 组织评审会议
3. 收集反馈并修订

## 编写要点

### 语言规范
- 使用主动语态
- 避免模糊词汇（"可能"、"大概"、"一些"）
- 使用具体数值而非相对描述
- 保持术语一致性

### 格式规范
- 使用标准Markdown格式
- 合理使用表格呈现结构化信息
- 添加目录便于导航
- 图文结合增强理解

### 内容规范
- 需求描述要SMART（具体、可衡量、可实现、相关、有时限）
- 每个功能都要有验收标准
- 异常场景要完整覆盖
- 业务规则要明确无歧义

## 质量检查

### 完整性
- [ ] 所有功能需求都有详细描述
- [ ] 非功能需求完整
- [ ] 数据字典完整
- [ ] 接口需求明确

### 清晰性
- [ ] 无模糊表述
- [ ] 术语使用一致
- [ ] 流程描述清晰
- [ ] 验收标准可测试

### 可行性
- [ ] 技术可行性已确认
- [ ] 资源需求合理
- [ ] 时间计划可行
- [ ] 风险已评估

### 一致性
- [ ] 功能间无冲突
- [ ] 与设计稿一致
- [ ] 与技术方案一致
- [ ] 版本号正确

## 常见问题避免

| 问题 | 避免方法 |
|------|----------|
| 需求描述过于笼统 | 使用具体场景和示例说明 |
| 缺少验收标准 | 每个功能必须有可测试的验收条件 |
| 业务规则遗漏 | 通过评审会议和边界测试发现遗漏 |
| 异常场景不全 | 使用负面测试思维，考虑各种异常情况 |
| 数据定义不清 | 提供数据字典，明确每个字段的含义和约束 |
| 性能要求不具体 | 给出具体数值，如"响应时间<200ms" |
| 范围蔓延 | 明确"不在范围内"的内容，变更走变更流程 |
