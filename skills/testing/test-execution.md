# 测试执行技能

## 概述

系统化地执行测试计划，包括测试用例执行、缺陷记录与跟踪、测试报告编写，确保测试过程规范、结果可追溯，为产品质量提供可靠的评估依据。

## 适用角色

- 测试工程师（主要）
- 质量保证工程师
- 开发工程师（自测）

## 输入

- **测试计划**：测试策略和范围定义
- **测试用例**：详细的测试用例文档
- **待测软件**：已部署的测试版本
- **测试环境**：配置好的测试环境

## 输出

- **测试执行记录**：用例执行结果
- **缺陷报告**：发现的缺陷详情
- **测试报告**：`docs/ccteam/04-testing/test-report.md`


### 缺陷级别定义

| 级别 | 名称 | 定义 | 示例 |
|------|------|------|------|
| P0 | 致命 | 系统崩溃、数据丢失、核心功能完全不可用 | 系统无法启动、支付失败但扣款 |
| P1 | 严重 | 核心功能异常、严重影响用户体验 | 登录失败、订单无法提交 |
| P2 | 一般 | 次要功能异常、有规避方案 | 搜索结果不准确、导出格式错误 |
| P3 | 轻微 | UI问题、体验问题、不影响功能 | 文案错误、对齐问题 |

### 缺陷生命周期

```
┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
│ Open │───▶│Assign│───▶│ Fixed│───▶│Verify│───▶│Closed│
└──────┘    └──────┘    └──────┘    └──────┘    └──────┘
                              │           │
                              │    ┌──────┤
                              │    │Reopen│
                              │    └──────┘
                              │
                        ┌─────▼─────┐
                        │Won't Fix/ │
                        │  Defer    │
                        └───────────┘
```

| 状态 | 说明 | 责任人 |
|------|------|--------|
| Open | 新创建的缺陷 | 测试工程师 |
| Assigned | 已指派给开发 | 测试/项目经理 |
| In Progress | 开发修复中 | 开发工程师 |
| Fixed | 开发已修复 | 开发工程师 |
| Verified | 验证通过 | 测试工程师 |
| Closed | 缺陷关闭 | 测试工程师 |
| Reopen | 验证不通过重新打开 | 测试工程师 |
| Won't Fix | 不予修复 | 产品/项目经理 |
| Deferred | 延期修复 | 产品/项目经理 |

## 执行步骤

### 步骤1：执行准备
1. 确认测试版本和环境
2. 检查环境可用性
3. 准备测试数据
4. 配置测试工具

### 步骤2：用例执行
1. 按优先级执行用例
2. 记录执行结果
3. 发现缺陷及时记录
4. 跟踪阻塞问题

### 步骤3：缺陷管理
1. 按规范提交缺陷
2. 跟踪缺陷状态
3. 验证修复结果
4. 更新缺陷状态

### 步骤4：进度跟踪
1. 每日总结执行情况
2. 更新测试进度
3. 沟通阻塞问题
4. 调整执行计划

### 步骤5：报告编写
1. 汇总执行数据
2. 分析缺陷数据
3. 评估质量风险
4. 给出发布建议
